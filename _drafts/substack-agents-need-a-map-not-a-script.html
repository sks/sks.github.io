Building an Enterprise AI Agent Platform in Go · #19

There's a moment in a great Rahman live set where the band stops replaying the film track and starts reading the room. Same score on paper. Completely different concert.

I've spent months teaching an AI SRE copilot to investigate real production incidents — across metrics, logs, warehouses, and messy human tickets. Same lesson every time:

If your agent only works when you ship it a forty-page bespoke runbook, the agent is not good yet. Not because runbooks are useless — because the product had no map underneath.

The runbook is a programmed track. The map is what a good investigator carries when the room changes. Here's what that map is — and when a runbook still earns its keep.

What the map is (four layers)

1. Topology at launch. Inject what discovery already knows — integrations, services, env tags, deploy repos — before the first tool call. No inventing dependency graphs mid-incident.

2. Verify-first probes. Branches write structured evidence (queue_depth_spike=true, crosscheck=mismatch). A deterministic verifier locks the root-cause class. The LLM drafts; code decides. No vibe-check judge.

3. Cross-plane reconciliation. Each datastore gets a turn; crosscheck must resolve to match, mismatch, or single_plane — never pending at gather complete. Inactive scoped signal + pod restart = no active incident on scoped path, not pod theater.

4. Verify-learn memory. Store operator verdicts and ruled-out paths. Recall them on the next similar alert.

Script vs map, in one line each:

  • Script: ship another notebook when the broker queue backs up.
  • Map: probe → discover → requery, with topology and memory built in.
  • Script: gate regex per tenant phrasing.
  • Map: verifier reads evidence keys.
  • Script: operator fixes RCA in Slack.
  • Map: verdict feeds the next run.

Per-tenant investigation stacks are symptoms of a copilot missing these four layers. When the map ships, those stacks should shrink — not vanish.

When runbooks still earn their keep

A live set still starts from the score. Runbooks stay valid for:

  • Human SOPs — on-call, auditors, training (PDF for people, not agent navigation).
  • Stable archetypes — known probe ladders encoded as map recipes, not rediscovered every run.
  • Reference demos — worked examples while discovery is still thin.
  • HITL remediation — who approves scale-down, which job to run (governance after evidence).
  • Local overlays — dashboard IDs, tag quirks, on top of topology — not instead of it.

The failure mode: runbook as sole dependency — forty pages per estate because the agent could not navigate alone.

Without the map

  • Wrong time window — 30 minutes when the signal needs 7 days.
  • Loudest spike wins over causal spike.
  • Beautiful RCA on a hollow gather stage.
  • "Observability gap" before exhausting tag discovery.
  • Sub-agents spin until spawn budget leaves pending_from_* stubs the gate accepts.

Operators need receipts — IDs, KPIs, ruled-out branches — not more narration.

Where scripts broke (two shapes)

Metric alerts — queue depth vs dependency-chain saturation: two parallel use cases, same failures. Narrow windows on sparse metrics. Redacted tokens pasted into SQL. Crosscheck stuck on pending. Gates cheering on empty text while ignoring real UUIDs. Presentation reading navigation JSON instead of gather evidence.

Symptom tickets — "Payments module stuck since Tuesday": keyword routing to the wrong workflow, question loops instead of probes, monitors picked by title match. Structure first; then tools.

Fixes were rarely more runbook poetry. They were topology, verifier, and gates — often encoding probe ideas the runbook already had.

Golden gates

Bring-up: deploy pipeline stages 1..N only; green each stage with a gate on tool outputs before adding N+1. (Hardware analogy.)

We owned two gate failures: regex wanted tenant GUID, agent correctly emitted a UUID; and a regex with negative lookahead that crashed Go's RE2 mid-run — caught by nothing in CI. Gates are code. Golden pass and fail fixtures, or you ship lies.

What to steal

  1. Topology at launch.
  2. Evidence → verifier → class.
  3. Crosscheck before present-RCA.
  4. Memory on verified verdicts.
  5. Bring-up one stage at a time.
  6. Test your gates like unit tests.
  7. Runbooks as overlay, not foundation.

Still rough: inactive incidents scored undetermined, path anchoring, chat paths that route but never investigate. All gate or contract bugs — fixable without a new foundation model.

Goal: an agent with a map, optionally guided by proven runbook recipes — not one that needs a fresh notebook per tenant.
